


PRD: DevFlow — 개발자의 사고 흐름 운영 시스템

---

## 1. 배경 및 문제 정의

### 1.1 현재 상황

Susan은 CJ대한통운 백엔드 개발자이자, 사이드 프로젝트를 Claude와 함께 기획부터 구현까지 수행하는 Heavy Claude 사용자다.

Claude CLI를 통해 코딩 생산성은 폭발적으로 증가했지만, 정작 **"무엇을 만들 것인가"를 정의하고 구조화하는 상위 레이어의 사고**는 여전히 체계 없이 이루어지고 있다.

### 1.2 핵심 문제

| 문제 | 구체적 상황 | 결과 |
| **사고의 휘발** | Claude와 나눈 기획/분석 대화가 채팅 로그 속에 묻힘 | 3일 뒤 돌아오면 맥락 소실, "뭐 얘기했더라?"부터 시작 |
| **단계 무인지** | 지금 이 프로젝트가 기획 단계인지, 기능 정의 단계인지, 구현 단계인지 본인도 모름 | 발산과 구현이 뒤섞여서 비효율 발생 |
| **지속성 실패** | 의지만으로 업무 흐름을 유지하려 하면 일주일도 안 감 | 사이드 프로젝트가 시작만 하고 중단되는 패턴 반복 |
| **컨텍스트 스위칭** | Claude 대화창 ↔ CLI ↔ FigJam ↔ 메모 등 도구 분산 | 사고의 연속성 단절, 몰입 파괴 |

### 1.3 근본 인사이트

> Claude의 대화 성능은 이미 충분하다.

> 병목은 "대화의 품질"이 아니라, **대화들이 프로젝트의 어떤 단계에서 어떤 결정을 만들어냈는가가 구조화되지 않는 것**이다.

### 1.4 Lesson Learned

- 하위 레이어(프롬프트 고도화, CLI 성능 개선)는 Anthropic이 이미 더 잘하고 있음
- **상위 레이어(업무 흐름의 추상화, 사고 단계의 시각화)에 집중**해야 함
- 사용자 의도를 더 잘 파악하는 것은 Anthropic의 목적 — 그걸 간과하고 직접 프롬프트를 고도화하려 했던 것이 이전 시도의 패착

## 2. 제품 비전

### 2.1 한 줄 정의

**DevFlow는 Claude와 함께 프로젝트를 수행하는 개발자의 "사고 흐름"을 시각화하고, 단계별로 구조화하여, 자연스러운 프로젝트 진행을 시스템이 이끌어주는 로컬 웹 애플리케이션이다.**

### 2.2 핵심 가치 제안

- AS-IS: 개발자가 Claude에게 "이거 만들어줘" → 코딩은 빠른데 방향은 불분명
- TO-BE: 시스템이 "지금 여기까지 왔어, 다음은 이 단계야" → 개발자가 판단하고 결정하는 사람으로 자연스럽게 전환
### 2.3 설계 철학

1. **거울 원칙**: 시스템은 Susan의 현재 위치를 비춰주는 거울이다. "너 지금 또 코더 모드야"를 인지시켜주는 것만으로 행동이 바뀐다.
1. **자연스러운 흐름**: 강제가 아니라, 다음 단계로 넘어가는 것이 가장 자연스러운 행위가 되도록 설계한다.
1. **상위 레이어 집중**: CLI의 기능을 고도화하지 않는다. CLI는 그대로 임베드할 뿐, 시스템의 가치는 "사고의 구조화"에 있다.

## 3. 타겟 사용자

### 3.1 페르소나: Susan (유일한 타겟)

- 30세 백엔드 개발자, Java/Oracle 3년차
- Claude CLI Heavy User — 기획, 분석, 코딩, 디버깅 전 과정에서 Claude 활용
- StrengthsFinder: Activator(시작 강함), Focus, Futuristic, Command, Ideation
- **패턴**: 아이디어 발산 → 흥분 → 며칠 뒤 맥락 소실 → 중단 또는 새 아이디어로 이동
- **핵심 니즈**: 사이드 프로젝트의 사고 흐름이 끊기지 않고 축적되는 환경
### 3.2 확장 가능 타겟 (제품화 시)

Susan과 같은 패턴을 가진 개발자:

- Claude/AI를 업무에 heavy하게 활용하는 1인 개발자
- 사이드 프로젝트를 시작은 잘 하지만 완주가 어려운 개발자
- "코더"에서 "문제 정의자"로 롤 전환을 시도하는 개발자

## 4. 사용자 시나리오

### 4.1 시나리오 A: 새 프로젝트 시작

1. Susan이 퇴근 후 DevFlow를 연다
1. "새 프로젝트" 생성 → 프로젝트명: "셀러 주문 자동화 도구"
1. 시스템이 기본 단계 파이프라인을 제시한다: [아이디어 발산] → [문제 정의] → [기능 구조화] → [기술 설계] → [구현] → [검증]
1. 현재 단계: "아이디어 발산"이 활성화됨
1. 우측에 Claude CLI 터미널이 열림
1. Susan이 떠오르는 생각을 Claude CLI로 자유롭게 대화
1. 대화 중 핵심 결정사항이 나오면 → "이 대화를 현재 단계의 결론으로 저장"
1. 충분히 발산했다고 느끼면 → "다음 단계로 이동"
1. "문제 정의" 단계가 활성화되며, 이전 단계의 요약이 컨텍스트로 자동 주입됨
### 4.2 시나리오 B: 3일 뒤 복귀

1. Susan이 3일 만에 DevFlow를 연다
1. 대시보드에 프로젝트 리스트가 보인다
1. "셀러 주문 자동화 도구" 클릭
1. 좌측: 각 단계별 노드가 보이고, 각 노드에 요약과 핵심 결정사항이 표시됨
  - [아이디어 발산] ✅ 완료 — "셀러들의 주문 처리 병목 해결"
  - [문제 정의] 🔄 진행중 — "멀티 채널 주문 통합이 핵심 문제"
  - [기능 구조화] ⬜ 대기
1. Susan이 "문제 정의" 노드 클릭 → 이전 CLI 세션 목록이 펼쳐짐
1. 마지막 세션 이어서 CLI 대화 가능 또는 새 세션 시작
1. 5분 안에 맥락 복구 완료, 바로 다음 작업 진행
### 4.3 시나리오 C: 단계 간 이동하며 작업

1. "기술 설계" 단계에서 Claude CLI로 아키텍처 논의 중
1. "구현" 단계로 이동 → CLI 세션이 새로 시작됨
1. Claude CLI로 코드 생성, 파일 수정 등 기존 CLI 기능 그대로 사용
1. 구현 중 "이 부분 설계를 다시 봐야겠다" → 좌측에서 "기술 설계" 노드 클릭
1. 해당 단계의 결정사항과 이전 CLI 세션 기록을 즉시 확인
1. 다시 "구현" 노드로 돌아와서 작업 계속
### 4.4 시나리오 D: 아이디어 Add-on

1. 구현 중 새로운 아이디어가 떠오름
1. "아이디어 발산" 단계에 새 세션을 추가 (이전 단계로 돌아가기)
1. 기존 구조에 새 아이디어가 합류
1. 시스템이 "이 아이디어가 기존 기능 정의에 영향을 줄 수 있어요" 알림
1. 관련 단계를 다시 검토하고 업데이트
1. 다시 구현으로 자연스럽게 복귀

## 5. 핵심 기능 정의

> 기능 비중: **사고 구조화(80%) + CLI 임베드(20%)**

### 5.1 [핵심] 프로젝트 파이프라인 뷰

**목적**: 프로젝트의 현재 위치를 한눈에 파악

- 프로젝트별 단계 파이프라인 시각화
- 기본 단계 템플릿 제공 (커스터마이징 가능): 아이디어 발산 → 문제 정의 → 기능 구조화 → 기술 설계 → 구현 → 검증/회고
- 각 단계의 상태 표시: 대기 / 진행중 / 완료
- 현재 활성 단계 하이라이트
- 단계 간 이동은 자유롭되, "다음 단계 이동" 시 현재 단계 요약을 유도
### 5.2 [핵심] 단계별 세션 관리

**목적**: 각 단계에서 이루어진 사고의 기록과 축적

- 각 단계 하위에 Claude CLI 세션 리스트
- 세션별 자동 요약 생성 (CLI 대화 내용 기반으로 로컬 파싱 또는 CLI를 통해 요약 요청)
- 핵심 결정사항(Decision Point) 마킹 기능
- 세션 간 시간순 타임라인 표시
- 이전 세션 컨텍스트를 새 세션에 자동 주입하는 옵션
### 5.3 [핵심] Claude CLI 패널

**목적**: 모든 단계에서 Claude와 상호작용하는 통합 인터페이스

- 우측 패널에 xterm.js 기반 Claude CLI 터미널 임베드
- CLI의 모든 기능을 그대로 사용 (대화, 코드 생성, 분석 등)
- API 사용 없음 — CLI가 이미 충분한 성능을 제공
- 단계별로 세션을 구분하여, 어떤 단계에서 어떤 CLI 대화가 이루어졌는지 기록
- 대화 내용은 시스템이 자동으로 캡처하여 해당 단계의 세션으로 저장
- 핵심 내용 "핀" 기능 → 해당 단계의 결정사항으로 저장
### 5.4 [핵심] 프로젝트 대시보드

**목적**: 복귀 시 5분 안에 맥락 복구

- 전체 프로젝트 리스트 + 각 프로젝트의 현재 단계
- 마지막 활동 시간 표시
- 프로젝트별 진행률 시각화 (단계 기준)
- 최근 결정사항 요약 미리보기
- "마지막으로 여기까지 했어요" 안내
### 5.5 [보조] CLI 세션 캡처 및 동기화

**목적**: CLI 대화 내용을 프로젝트 구조에 자동 연결

- CLI 터미널의 입출력을 실시간 캡처하여 현재 활성 단계의 세션으로 저장
- 세션 전환 시 CLI 컨텍스트 유지 (이전 세션 이어서 대화 가능)
- CLI 출력 중 코드 블록, 분석 결과 등을 자동 감지하여 태깅
- 별도의 CLI 고도화 기능은 없음 — 있는 그대로 제공
### 5.6 [보조] 단계 전환 가이드

**목적**: 자연스러운 흐름 유도 (강제성의 부드러운 구현)

- 다음 단계로 이동 시 → 현재 단계 요약 작성을 유도
- 이전 단계로 되돌아가는 것도 자유롭되, 그 사유를 기록하도록 유도
- "구현 단계인데 아직 기능 정의가 안 됐어요" 같은 경고
### 5.7 [보조] 사고 타임라인

**목적**: 전체 프로젝트의 사고 흐름을 시간축으로 조망

- 프로젝트 전체 기간의 타임라인 뷰
- 언제 어떤 단계에서 어떤 세션이 있었는지 시각화
- 결정사항들의 시간순 나열
- "이 프로젝트는 이런 흐름으로 진화해왔구나"를 한눈에 파악

## 6. 기술 아키텍처

### 6.1 시스템 구성

```javascript
DevFlow (localhost:3000)

┌──────────────────┬──────────────────────────┐
│   Next.js App    │   Next.js API Routes     │
│   (Frontend)     │   (Backend)              │
│                  │                          │
│   Dashboard      │   Prisma ORM             │
│   Pipeline       │   ├ 현재: SQLite         │
│   CLI Panel      │   └ 확장: Supabase       │
│                  │                          │
│                  │   xterm.js + node-pty    │
│                  │   (Claude CLI 임베드)     │
└──────────────────┴──────────────────────────┘
         │
         │ Tailscale / Cloudflare Tunnel
         ▼
      모바일 (외부 접근)
```

### 6.2 기술 스택

| 영역 | 기술 | 선정 이유 |
| **프론트엔드** | Next.js 14 (App Router) + TypeScript | SSR, API Routes 통합, 빠른 개발 |
| **UI** | Tailwind CSS + shadcn/ui | Claude와 UI 빠르게 구현 가능 |
| **상태관리** | Zustand | 가볍고 심플, 로컬 앱에 적합 |
| **DB** | SQLite (Prisma ORM) | 로컬 우선, Prisma를 통해 Supabase(PostgreSQL) 전환 무중단 가능 |
| **Claude 연동** | xterm.js + node-pty (CLI 임베드 only) | API 비용 없음, CLI의 기존 성능을 그대로 활용 |
| **터미널** | xterm.js + node-pty | 브라우저 내 실제 터미널 임베드 |
| **원격 접근** | Tailscale 또는 Cloudflare Tunnel | 모바일에서 안전하게 접근 |
| **마크다운** | react-markdown + remark | 대화 내용, 요약 렌더링 |

### 6.3 원격 접근 옵션 비교

| 방식 | 장점 | 단점 | 추천 상황 |
| **Tailscale** | VPN 기반, 매우 안전, 설정 간단 | 모든 기기에 Tailscale 설치 필요 | Susan 개인 기기에서만 접근 |
| **Cloudflare Tunnel** | 무료, 설치 없이 URL로 접근 | 도메인 필요, 인증 별도 설정 | 어디서든 빠르게 접근 |
| **Tailscale + Funnel** | Tailscale의 보안 + 외부 URL 제공 | Tailscale 계정 필요 | 두 가지 장점을 모두 원할 때 |

### 6.4 데이터 모델 (Prisma ORM — 현재 SQLite, 확장 시 Supabase/PostgreSQL)

| 테이블 | 주요 필드 | 역할 |
| **projects** | id, name, description, status, created_at, updated_at | 프로젝트 관리 |
| **stages** | id, project_id, name, order_index, status, summary | 단계 (파이프라인 노드) |
| **sessions** | id, stage_id, title, auto_summary | 대화 세션 |
| **messages** | id, session_id, role, content, is_pinned | 대화 메시지 |
| **decisions** | id, stage_id, session_id, content, context | 결정사항 (Decision Points) |
| **activities** | id, project_id, stage_id, activity_type, description | 프로젝트 활동 타임라인 |

### 7.2 대시보드 (프로젝트 리스트)

## 9. 성공 기준

### 9.1 정량적 기준

| 지표 | 현재 (AS-IS) | 목표 (TO-BE) |
| 프로젝트 복귀 시 맥락 복구 시간 | 30분+ (이전 대화 뒤지기) | 5분 이내 |
| 사이드 프로젝트 완주율 | ~20% (시작 후 중단 패턴) | 60% 이상 |
| 사고 단계 인지도 | 없음 (뒤섞여 있음) | 항상 현재 단계 인지 |
| 컨텍스트 스위칭 횟수 (도구 간) | 5+개 도구 | DevFlow 1개로 통합 |

### 9.2 정성적 기준

- "아, 이 프로젝트 여기까지 와있었구나"가 열자마자 보인다
- 구현 단계에서 자연스럽게 "이건 기획 단계에서 다시 생각해봐야겠다"가 가능하다
- 3일 뒤에 돌아와도 이어서 작업하는 게 부담이 아니라 자연스럽다
- "오늘은 기능 구조화 단계를 마무리하자"처럼 목표가 명확해진다

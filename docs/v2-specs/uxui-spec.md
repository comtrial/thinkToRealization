## 0. 전문가 팀 구성 및 회의 기록

> 이 설계서는 다음 5인의 전문가 에이전트가 심층 토론을 거쳐 도출했습니다. 각 의사결정에 대해 역할별 관점이 충돌하는 지점을 명시하고, 합의 과정을 기록합니다.

**참여자:**
- **Mira** — Node 기반 시각화 UI 전문가. xyflow/React Flow 생태계에 정통하며, 노드 밀도·시인성·인터랙션 마이크로 패턴에 집중
- **Joon** — Linear UX 수석 전문가. Linear의 keyboard-first 철학, 정보 밀도, surface hierarchy, 네비게이션 모델 전문
- **Karri** — Linear CEO 역할. "왜 이 기능이 있어야 하는가?"를 끊임없이 질문. 불필요한 복잡성 제거와 제품 비전 정합성 판단
- **Dev (개발자)** — Vibe coding 전문 창립 개발자. Claude CLI 기반 실제 개발 흐름에 정통하며, 무엇이 실제로 유용하고 무엇이 허상인지 냉철하게 판단
- **PM** — UX의 추상적 영역을 개발 가능한 구체적 스펙으로 변환하는 전문가. 누락 방지 및 잘못된 설계 탐지 역할

## 1. 전문가 팀 심층 회의록

### 1.1 Round 1: "캔버스가 정말 메인 화면이어야 하는가?"

**Karri (CEO):** "Linear가 성공한 이유는 리스트 뷰가 가장 빠르게 정보를 스캔할 수 있기 때문이다. 캔버스를 메인으로 놓으면 매번 줌인/줌아웃 오버헤드가 생긴다. 정말 캔버스가 첫 화면이어야 하나?"

**Dev (개발자):** "동의한다. 내가 vibe coding으로 실제 개발할 때, 앱을 열면 '지금 뭘 해야 하지?'가 가장 먼저 떠오른다. 캔버스 전체를 보고 싶은 건 기획 초반뿐이다. 작업 중에는 '현재 In Progress인 노드'만 빠르게 접근하고 싶다."

**Mira (시각화 전문가):** "하지만 기획안의 핵심이 '사고의 공간적 배치'다. Welcome Back 대시보드를 진입점으로 두되, 캔버스로의 전환이 0-friction이면 해결된다."

**Joon (Linear UX):** "Linear도 Inbox → Issue Detail 흐름이 대부분이다. 메인 화면은 '오늘 해야 할 일'의 스냅샷이고, 캔버스는 '전체 그림이 필요할 때' 진입하는 구조가 맞다."

**PM:** "정리하면, 진입점은 Welcome Back 대시보드(Linear의 My Issues와 유사), 캔버스는 상시 접근 가능한 두 번째 탭. 이 구조가 기획안의 3-Step 루프를 깨뜨리지 않는지 검증 필요."

> **합의:** 앱 진입 시 Welcome Back 대시보드 → 캔버스는 상단 탭 전환 (Cmd+1: 대시보드, Cmd+2: 캔버스). 대시보드에서 노드 클릭 시 사이드패널 직접 오픈.

---

### 1.2 Round 2: "노드 카드 디자인 — 정보 밀도 vs 시인성"

**Mira:** "xyflow 커스텀 노드는 React 컴포넌트이므로 자유도가 높다. 문제는 캔버스에서 20~30개 노드가 동시에 보일 때 가독성. 노드 안에 너무 많은 정보를 넣으면 줌 레벨에 따라 읽을 수 없다."

**Joon:** "Linear의 보드 카드를 봐라. 타이틀 + 상태 아이콘 + 아사인 아바타, 이 3가지만 카드에 표시한다. 나머지는 hover 또는 클릭으로 펼친다. 노드도 동일해야 한다."

**Dev:** "실제 사용에서 가장 중요한 건 '이 노드가 지금 활성인지, 완료인지, 대기인지'. 세션 수나 결정사항 수는 사이드패널에서 보면 충분하다."

**Karri:** "노드는 두 가지 줌 레벨 상태를 가져야 한다. 축소 시 → 아이콘 + 타이틀 + 상태 색상. 확대 시 → 추가 메타 정보 표시. Semantic zoom."

**PM:** "기획안에는 '세션 N개, 결정 N개, 변경 파일 N개'가 노드에 표시된다고 되어있다. 이걸 두 단계 줌으로 분리하면 기획안을 만족시키면서 시인성도 유지된다."

> **합의:** 노드 카드는 2-Level Semantic Zoom. Level 1 (줌아웃): 아이콘 + 타이틀 + 상태 도트. Level 2 (줌인): 설명 + 세션/결정/파일 카운터 추가. 활성 세션은 어떤 줌에서든 좌상단 초록 도트로 표시.

---

### 1.3 Round 3: "사이드패널 설계 — Notion Side Peek를 어떻게 적용할 것인가?"

**Joon:** "Notion의 Side Peek는 '데이터베이스 행의 상세 보기'를 리스트 옆에 표시하는 패턴이다. DevFlow에서는 '캔버스 노드의 상세 보기'를 캔버스 옆에 표시하는 것이 직접적인 대응이다."

**Mira:** "캔버스 우측에 고정 사이드패널을 두면, 캔버스 영역이 좁아져서 노드 그래프 가독성이 떨어진다. 패널 너비는 최소 400px, 최대 50% 이하로 제한해야 한다."

**Dev:** "사이드패널 안에서 가장 많이 하는 행위는 CLI 세션이다. 터미널은 최소 너비가 있어야 사용 가능하다. 400px이면 80컬럼 터미널이 들어간다. 그 이하는 쓸 수 없다."

**Karri:** "패널은 3가지 모드를 지원해야 한다: 닫힘 → Side Peek (40%) → Full Page (전체 화면). Linear도 Issue를 Side Peek 또는 Full Page로 열 수 있다."

**PM:** "기획안에서 '노드 클릭 → 사이드 패널에 Claude CLI 터미널 열림'이라고 명시되어 있다. 패널 구조를 구체적으로 정의해야 한다."

> **합의:** 사이드패널 3모드: Closed → Side Peek (우측 40%, 최소 400px) → Full Page (오버레이). 노드 클릭 시 기본 Side Peek. Cmd+Enter로 Full Page 전환. 패널 내부는 탭 구조: Overview | Sessions | Files.

---

### 1.4 Round 4: "CLI 터미널을 사이드패널에 넣는 것이 맞는가?"

**Dev:** "이건 핵심 논쟁이다. 사이드패널에 터미널을 넣으면 '사고 도구'와 '실행 도구'가 하나의 패널에 공존한다. 기획안의 Step 2 '하기'는 사이드패널 내에서 수행되는데, 노드 개요와 터미널이 어떻게 공존하나?"

**Mira:** "터미널은 패널의 하단 60%를 차지하고, 상단 40%에 노드 개요(상태, 결정사항)가 있는 split-view가 자연스럽다. 터미널이 비활성이면 개요가 전체를 채운다."

**Joon:** "Linear는 Issue 상세에서 Description → Activity → Comments가 수직 스크롤로 나열된다. DevFlow도 '노드 메타 → 결정사항 → 세션 목록' 순서로 스크롤하다가, 세션을 클릭하면 터미널이 확장되는 구조가 낫다."

**Karri:** "터미널이 항상 보이면 안 된다. 세션을 시작하거나 이어갈 때만 터미널 영역이 등장해야 한다. '사고 도구에서 필요할 때만 실행 도구를 꺼낸다'는 원칙."

**Dev:** "맞다. 터미널은 접힌 상태가 기본이고, '[+ 새 세션]' 또는 '[이어서 대화]'를 누르면 패널 하단에서 슬라이드업으로 등장. 이건 VS Code의 통합 터미널 패턴과 동일하다."

**PM:** "기획안에서 터미널은 '사이드 패널에서 열린다'고만 되어 있고 구체적 레이아웃은 없다. 이 설계를 확정해야 한다."

> **합의:** 사이드패널 = 상단 Overview 섹션 (노드 메타, 결정사항, 세션 리스트) + 하단 Terminal 섹션 (기본 접힘, 세션 활성화 시 슬라이드업 확장). 터미널과 Overview 경계는 드래그로 리사이즈 가능. 터미널 없이도 Overview만으로 노드를 관리할 수 있어야 한다.

### 1.5 Round 5: "노드 타입별 차별화 — 모든 타입이 정말 필요한가?"

**Karri:** "6가지 노드 타입이 있다: Idea, Decision, Task, Issue, Milestone, Note. MVP에서 6개 전부 만들면 개발 비용이 크다. 핵심만 남겨라."

**Dev:** "실제 vibe coding에서 내가 쓰는 사고 단위는 3가지다. '해야 할 일(Task)', '결정한 것(Decision)', '떠오른 것(Idea)'. Issue는 Task의 변종이고, Milestone은 그룹핑이고, Note는 어디든 붙는 메모다."

**Joon:** "Linear는 Issue라는 단일 타입에 Label로 구분한다. 타입을 적게 가져가고 태그로 구분하는 게 더 유연하다."

**Mira:** "하지만 시각적으로 캔버스에서 타입이 즉시 구분되어야 한다. 아이콘 + 색상으로 타입을 구분하면 6개도 충분히 시인성을 유지할 수 있다."

**PM:** "기획안에 6가지가 명확히 정의되어 있다. MVP에서 일부를 빼면 기획안과 불일치. 하지만 구현은 단계적으로 가능. v2.0에서 Task + Idea + Decision 3개, v2.1에서 Issue + Note + Milestone 추가."

> **합의:** 노드 타입은 6개 유지 (기획안 존중). 단, MVP(v2.0)에서는 Task, Idea, Decision 3개 먼저 구현. 내부 데이터 모델은 6개 모두 지원하되, UI 렌더링만 단계적 추가. 시각적 구분은 좌측 수직 컬러 바 + 아이콘으로 처리.

---

### 1.6 Round 6: "디자인 톤 — Linear 참고하되 어둡지 않게"

**Joon:** "Linear의 최근(2025) 리디자인은 monochrome에서 더 중립적이고 타임리스한 방향으로 갔다. 다크 모드가 기본이지만, 라이트 모드도 충분히 세련됐다."

**Karri:** "Susan의 요구는 '너무 어두침침하지 않게'. 라이트 모드를 기본으로 하되, 다크 모드도 지원. 라이트 모드에서의 색상 시스템이 핵심이다."

**Mira:** "캔버스 배경은 너무 순백이면 눈이 피곤하다. Linear 라이트 모드처럼 약간의 warm gray (예: #FAFAF9 또는 #F8F8F7)를 배경으로. 노드 카드는 흰색, 엣지는 gray-300 수준."

**Dev:** "터미널 영역만큼은 어두운 배경이 자연스럽다. 코드/CLI는 다크 배경이 관행이므로, 사이드패널 터미널 섹션만 dark theme."

**PM:** "정리하면: 전체 앱은 Linear Light 기반 warm neutral, 터미널 섹션만 dark. 색상 시스템은 LCH 기반으로 surface elevation을 관리."

> **합의:** 기본 테마 = Warm Light (배경 #FAFAF9, 표면 #FFFFFF, 보더 #E5E5E3). 액센트 = Indigo-600 (#4F46E5). 상태 색상: Backlog(gray), Todo(amber), InProgress(blue), Done(green). 터미널 = Dark 배경 (#1E1E1E). 다크 모드는 v2.2 이후.

### 1.7 Round 7: "하이라이트와 노드 승격 — 실제로 쓸까?"

**Dev:** "솔직히 말하면, vibe coding 중에 Claude 응답 하나하나에 별을 클릭할 여유가 없다. 대화가 빠르게 진행되는데 멈춰서 '이게 중요한 결정이다'를 마킹하는 건 흐름을 깨뜨린다."

**Karri:** "그러면 언제 하이라이트를 하나?"

**Dev:** "세션이 끝나고, '방금 뭘 결정했지?'를 정리할 때. 즉, 실시간이 아니라 회고(retrospective) 시점이다. 세션 종료 후 로그를 스크롤하며 별을 찍는 게 현실적이다."

**Mira:** "그러면 별은 세션 활성 중에도 되고, 종료 후에도 되어야 한다. 세션 기록 보기 모드에서도 별 가능."

**PM:** "기획안에서는 '세션 활성 중 Claude 응답에 별 클릭'으로 명시되어 있다. 세션 종료 후에도 가능하도록 확장하면 기획안을 포함하면서 현실적 사용도 커버한다."

**Joon:** "노드 승격(별 → 새 노드)은 중요한 기능이지만 Discovery가 어렵다. 컨텍스트 메뉴에 넣되, 처음 사용 시 툴팁으로 안내."

> **합의:** 하이라이트는 세션 활성 중 + 세션 기록 열람 중 모두 가능. 노드 승격은 하이라이트 항목의 hover 시 표시되는 액션 버튼. 첫 사용 시 코치마크(tooltip) 표시.

### 1.8 Round 8: "캔버스 위의 '논의 영역 시각화' — 프레임 설계"

**Mira:** "기획안에서 '프레임 = 단계별 그룹핑 (FigJam의 Section)'이라고 했다. xyflow에서는 Group Node로 구현 가능하다. 부모-자식 노드 관계로 드래그 시 함께 이동."

**Dev:** "프레임이 너무 많으면 캔버스가 복잡해진다. 사용자가 자유롭게 만들되, 자동으로 제안하지는 말아야 한다."

**Karri:** "프레임은 선택적 도구다. 사용하지 않아도 노드만으로 충분히 작업 가능해야 한다."

**PM:** "프레임의 시각적 표현: 반투명 배경 + 점선 테두리 + 상단 라벨. 노드를 프레임 안으로 드래그하면 자동 귀속. FigJam Section과 동일한 패턴."

> **합의:** 프레임 = xyflow Group Node. 반투명 배경(#F5F5F4 50% opacity), 점선 보더, 좌상단 라벨. 노드 드래그 시 프레임 안에 들어가면 자동 귀속. 프레임 자체도 드래그/리사이즈 가능. MVP에서 포함하되 최소 기능만.

## 2. UXUI 상세 정의서

### 2.1 전체 레이아웃 구조

```
+-------------------------------------------------------------+
|  Header Bar (48px)                                          |
|  [= 사이드바 토글] [DevFlow] [프로젝트: Sequeliquance v]     |
|  -------- [대시보드] [캔버스] -------- [Cmd+K] [설정]        |
+-------+---------------------------------+-------------------+
| Side  |                                 |  Side Panel       |
| bar   |  Main Content Area              |  (사이드 피크)     |
| (220px|  - 대시보드 또는 캔버스 -         |  (0 / 400~50%)   |
| 접힘  |                                 |                   |
| 가능) |                                 |  +- Overview ---+ |
|       |                                 |  | 노드 메타     | |
| My    |  +----------------------+       |  | 결정사항      | |
| Work  |  |  캔버스/대시보드 컨텐츠|       |  | 세션 리스트   | |
| ----- |  |                      |       |  +- Terminal ---+ |
| Inbox |  |                      |       |  | (접힘/확장)   | |
| 프로젝|  |                      |       |  | xterm.js      | |
| 트 목록|  |                      |       |  |              | |
|       |  +----------------------+       |  +--------------+ |
| ----- |                                 |                   |
| [+프로|  [줌: - . +] [핏] [미니맵]       |                   |
| 젝트] |                                 |                   |
+-------+---------------------------------+-------------------+
```

**레이아웃 핵심 원칙:**

## 3. PRD (Product Requirements Document)

### 3.1 제품 정의

- **제품명:** DevFlow v2
- **한줄 정의:** Claude CLI 세션을 최하단 기반으로, 기능 단위의 사고를 캔버스에서 시각화하고 관리하는 개발 사고 운영 도구
- **대상 사용자:** Vibe coding을 통해 Claude CLI로 개발하는 1인 개발자
- **핵심 가치:** 복귀 시 30초 내 맥락 복구, 사고 흐름의 구조적 시각화, 상태 관리의 자동화

### 3.2 핵심 사용자 스토리

**US-1: 앱 진입 시 맥락 복구**
> "개발자로서, 앱을 열면 현재 진행 중인 작업과 다음 할 일을 즉시 파악하여 맥락 복구 시간을 최소화하고 싶다."
- 수용 기준: 대시보드에서 In Progress 노드 목록 즉시 표시, 클릭 1번으로 세션 재개 가능

**US-2: 캔버스에서 사고 흐름 시각화**
> "개발자로서, 프로젝트의 전체 사고 흐름을 캔버스에서 공간적으로 파악하여, 의존 관계와 진행 상황을 한눈에 보고 싶다."
- 수용 기준: 노드 + 엣지로 DAG 표현, 상태별 시각적 구분, 프레임으로 그룹핑

**US-3: 노드 안에서 Claude CLI 세션 실행**
> "개발자로서, 노드를 클릭하면 사이드패널에서 Claude CLI 세션을 시작/재개하여, 사고 단위와 실행 단위를 연결하고 싶다."
- 수용 기준: 사이드패널 터미널에서 PTY 실행, claude --resume 지원, 세션-노드 1:N 매핑

**US-4: 결정사항 기록 (하이라이트)**
> "개발자로서, Claude 응답 중 중요한 결정사항을 1클릭으로 기록하여, 나중에 왜 이 결정을 했는지 추적하고 싶다."
- 수용 기준: 별 클릭 즉시 decisions 배열에 추가, 세션 활성/비활성 모두 가능, 노드 승격 지원

**US-5: 자동 상태 반영 (Track A)**
> "개발자로서, 세션 시작/종료 시 노드 상태가 자동으로 변경되어, 수동 상태 관리에 드는 인지 부담을 줄이고 싶다."
- 수용 기준: 세션 시작 → In Progress 자동, 세션 종료 → 프롬프트 (완료/이어서), 모든 전이 결정적

**US-6: 세션 이어가기**
> "개발자로서, 이전에 중단한 Claude 세션을 이어가서, 대화 맥락을 유지한 채 작업을 계속하고 싶다."
- 수용 기준: claude --resume SESSION_ID 실행, 세션 기록에서 [이어서 대화] 1클릭

**US-7: 노드 간 관계 표현**
> "개발자로서, 노드 간의 의존/순방향/관련/회귀/분기 관계를 엣지로 표현하여, 사고의 구조를 명확히 하고 싶다."
- 수용 기준: 5가지 엣지 타입, 핸들 드래그로 생성, 우클릭으로 타입 변경

## 4. 검증 단계: 기획안 대비 UXUI 설계 정합성 체크

### 4.1 기획안 핵심 원칙 vs 설계 반영 여부

| 기획안 원칙 | 설계 반영 | 상태 |
|------------|---------|------|
| "사고 캔버스가 주인공, 터미널은 실행 도구" | 캔버스 = 메인 뷰 (Cmd+2), 터미널 = 사이드패널 하단 접힘 | 반영 |
| "구조화된 규격이 자유보다 낫다" | 6개 노드 타입 + 5개 엣지 타입 + 상태 워크플로우 고정 | 반영 |
| "프리핸드 드로잉 없음" | xyflow 사용, 드로잉 기능 미포함 | 반영 |
| "3-Step 핵심 루프: 보기→하기→관리하기" | 대시보드/캔버스(보기) → 사이드패널(하기) → Track A(관리하기) | 반영 |
| "모든 트리거는 이진(binary)" | Track A: 세션 이벤트만. Track B: 사용자 클릭만. AI 없음 | 반영 |
| "Layer 0: 세션 이벤트 버스" | FR-04로 구현. 4개 이벤트 정의 | 반영 |
| "노드당 활성 세션 최대 1개" | 엣지케이스 Case 1로 명시 | 반영 |
| "기본값 = 미완료" | 종료 프롬프트 3초 후 fade out, 무시 = 상태 유지 | 반영 |
| "1 action = 1 click" | 모든 Track B 액션이 1 클릭/단축키로 완료 | 반영 |
| "Cmd+K 커맨드 팔레트" | FR-06, cmdk 라이브러리 | 반영 |
| "키보드 퍼스트 + UI 병행" | 모든 액션에 단축키 + UI 버튼 병렬 제공 | 반영 |
| "Linear의 Issue 상태 워크플로우" | Backlog→Todo→In Progress→Done 동일 | 반영 |
| "Triage 인박스" | 사이드바 Inbox + 캔버스 Inbox 영역 | 반영 |
| "Obsidian 호환 md 파일" | 데이터 모델에 md 파일 경로 포함. v2.2 구현 | 계획 |

### 4.2 기획안에 있으나 설계에서 변경/확장된 부분

| 기획안 내용 | 설계 변경 | 변경 이유 |
|------------|---------|---------|
| "캔버스가 메인 화면" | Welcome Back 대시보드 → 캔버스 탭 구조 | Karri+Dev 의견: 매 진입 시 캔버스 전체를 볼 필요 없음. 대시보드가 더 빠른 맥락 복구 |
| "노드 카드에 세션N개, 결정N개, 파일N개 표시" | Semantic Zoom 2단계로 분리 | Mira 의견: 줌아웃 시 작은 텍스트 가독성 문제 |
| "사이드패널에 CLI 터미널 열림" | 터미널 기본 접힘 + 슬라이드업 확장 | Dev 의견: 터미널이 항상 보이면 "사고 도구"보다 "실행 도구"로 인식됨 |
| "하이라이트는 세션 활성 중" | 세션 활성 + 세션 기록 열람 중 모두 가능 | Dev 의견: 실시간보다 회고 시점에 마킹하는 게 현실적 |
| 6개 노드 타입 전부 v2.0 | MVP에서 Task, Idea, Decision 3개 먼저 | PM 의견: 개발 비용 조절. 데이터 모델은 6개 모두 지원 |

### 4.3 설계 방향 올바름 검증

**검증 기준 1: "코드가 아닌 의사결정 레이어에서 프로젝트 운영"**
→ 반영됨. 노드 카드에 코드는 없다. 타이틀, 상태, 결정사항, 세션 카운터만 표시. 코드 diff는 Files 탭에서만 접근.

**검증 기준 2: "복귀 시 30초 내 맥락 복구"**
→ 반영됨. Welcome Back 대시보드에서 In Progress 노드 즉시 표시 + 1클릭 세션 재개.

**검증 기준 3: "시스템 행동 100% 예측 가능"**
→ 반영됨. Track A (결정적 이벤트 리스너) + Track B (사용자 클릭). AI 추론 없음. 모든 전이 규칙 테이블로 명시.

**검증 기준 4: "Linear 핵심 가치 흡수"**
→ 반영됨. Cmd+K, 키보드 퍼스트, 상태 워크플로우, Inverted L-shape 레이아웃, 사이드바 접기.
→ 미적용: 팀/멀티유저, 리스트/보드 뷰, Git 연동 (기획안과 일치).

**검증 기준 5: "디자인이 너무 어둡지 않음"**
→ 반영됨. 기본 테마 Warm Light (#FAFAF9). 다크 영역은 터미널 섹션만.

**검증 기준 6: "xyflow 선택의 타당성"**
→ 반영됨. 커스텀 React 노드 6종, 엣지 5종, Group Node(프레임), dagre 자동 레이아웃 모두 xyflow 네이티브 지원.
